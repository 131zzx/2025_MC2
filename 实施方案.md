# VAST Challenge 2025 MC2 · 可视化系统实施方案

项目目标：为记者 Moray 构建一个可交互的视觉分析系统，通过对 FILAH、TROUT、journalist 三个数据集的对比分析，揭示 COOTEFOO 委员会的真实行为与各方数据采样偏见。

---

## 一、调查逻辑链

```
FILAH（渔业方）持有部分记录 ─┐
                              ├─→ 各自得出「委员会偏袒对方」的结论
TROUT（旅游方）持有部分记录 ─┘
                              ↓
          Moray 获得全量记录（journalist.json）
                              ↓
  可视化系统揭示：
    ① 各数据集本身的采样偏见（Sampling Bias）
    ② 委员会成员的真实行为（Behavioral Pattern）
    ③ 缺失数据如何扭曲外界对委员的判断
```

**最终交付物**：
1. 可交互的 Web 可视化系统（工程）
2. 对四个官方问题的分析结论（每题配图 + 简短文字，填入 Answer Sheet）
3. 可复现的数据预处理脚本（Python）

## 三、数据语义模型

### 3.1 知识图谱层次结构

```
Meeting（会议）
  └─ part_of ─→ Discussion / Plan（讨论 / 计划）
        ├─ about / plan ─→ Topic（议题）
        ├─ travel ─→ Place（目的地）
        └─ participant ─→ Person / Organization
                              ↑ sentiment[-1,1], industry[], reason

Trip（实际行程，独立链条）
  ├─ → Person（出行人，无 role 字段）
  └─ time + → Place（途经地点，按时间戳排序）
```

|----------|---------|---------|-----------|

| | `new_crane_lomark` | Lomark 新起重机资金 | −1 |

### 3.3 文本字段 → 标量的完整转换策略

知识图谱中大量信息以文本形式存储，需系统性转化为可视化用的数值标量：

| 文本字段 | 所在位置 | 转换方法 | 输出标量 |
|---------|---------|---------|---------|
| 节点 ID（如 `expanding_tourist_wharf_Meeting_3`） | 所有节点 | 正则提取 topic 前缀 → 查产业表 | bias_score ∈ {−1, 0, +1} |
| `sentiment` | participant 边 | 直接使用 | float ∈ [−1, +1] |
| `industry` 列表（如 `["tourism"]`） | participant 边 | 独热编码后统计占比 | fishing_ratio / tourism_ratio |
| `reason` 文本 | participant 边 | 关键词分类（"supports"→正，"opposes"→负）or 直接用 sentiment | 归入对应 industry 的情感统计 |
| `zone` 字符串 | place 节点 | 有序映射：`tourism`→+1，`industrial`→−1，`commercial/residential`→0 | zone_score |
| `plan_type` 字符串 | plan 节点 | 类别计数（Travel/Report/proposal 各多少） | 活动类型分布向量 |
| `date` 字符串 | meeting/trip 节点 | 解析为时间序列索引 | 时间轴坐标 |
| 节点 `type` 字符串 | 所有节点 | 枚举编码（8 种类型 → 整数 ID） | 节点类型分类轴 |

**核心偏见指数公式**：

```
bias_index(D) = (fishing活动数 − tourism活动数) / (fishing + tourism活动数)
```

- 值域 [−1, +1]
- **+1** = 完全偏旅游；**−1** = 完全偏渔业；**0** = 均衡
- 可对每个**数据集**、每个**成员**、每次**会议**分别计算
- 也可加权：用 `sentiment` 值替代活动计数，得到**情感加权偏见指数**

**人员覆盖率公式**：

```
coverage(person, D) = |{activities of person in D}| / |{activities of person in journalist}|
```

- FILAH 中 Ed Helpsford、Teddy Goldstein、Tante Titan 的 coverage = 0%

## 四、四个问题的完整分析方案

### Q1：各方指控是否被其自身数据集支持？

**使用数据**：仅 `FILAH.json` + `TROUT.json`，**不使用** journalist。

**分析层次**：

| 层次 | 回答什么 | 核心证据字段 |
|------|----------|------------|
| A. 数据集采样偏见 | 记录本身是否片面？能否支撑该阵营的整体叙事？ | 人员覆盖率、节点类型构成、trip/meeting 比例 |
| B. 成员行为偏见 | 该数据集内是否可见成员对渔/游议题的系统性倾向？ | participant 的 industry、sentiment、reason |

#### Q1-A：FILAH 数据能否支持「委员会偏袒旅游」的指控？

若指控被自身记录支持，应看到：委员会成员参与旅游 topic 的记录占多数；participant 边中 tourism industry 比例高；sentiment 对旅游议题整体偏正向。

**预期发现**：
- 仅 3/6 成员 → 无法代表「委员会整体」偏向
- trip 节点占 48% → 数据形态偏「行程日志」而非会议记录
- 需分析 3 名有记录成员（Seal、Simone Kat、Carol Limpet）各自的议题分布

**Q1-A 图表**：
1. FILAH 议题产业占比（饼图/条形图，渔/游/混合/中性各几个 discussion/plan）
2. 6 人参与矩阵（横轴成员，纵轴议题，有色格=有记录，灰格=无记录）
3. 3 名有记录成员的 participant：fishing vs tourism 情感分布（分组条形）
4. FILAH trip 落点 zone 分布 vs discussion 产业分布（对比双条形）

#### Q1-B：TROUT 数据能否支持「委员会偏袒渔业」的指控？

若指控被自身记录支持，应看到：渔业 topic 的讨论/计划占多数；委员会成员对渔业 topic 的 sentiment 更为正向；旅游 topic 的参与相对较少。

**预期发现**：
- 6/6 成员均有记录 → 适合做成员行为偏见分析
- trip 仅 18 条 → 叙事集中在会议室，地理行为证据弱
- participant 约 94 条 → 可逐人统计 industry/sentiment

**Q1-B 图表**：
1. TROUT 议题产业占比（与 Q1-A 同类型图，便于对比）
2. 成员 × 产业 热力图（行=6 名成员，列=渔/游/混合，颜色=活动次数）
3. sentiment 按 topic 产业分类箱线图（渔业议题 vs 旅游议题的情感分布）
4. 活动类型构成堆叠条形（meeting/discussion/plan/trip 各有多少条）

#### Q1-C：两数据集偏见对比

**Q1-C 图表**：
1. 雷达图并排：人员覆盖率 / trip 密度 / 渔业议题占比 / 旅游议题占比（FILAH vs TROUT）
2. 节点类型构成堆叠条形（FILAH trip 峰 vs TROUT discussion 峰）
3. 同一 6 人在两库中活动量分组条形（可视化「被遗漏」的对比）

**Q1 结论框架**：

| 数据集 | 自身数据是否支持己方指控？ | 成员行为偏见证据？ |
|--------|--------------------------|-----------------|
| FILAH | 部分支持（有旅游活动记录），但人员缺口严重，无法代表整体 | 仅对 3 人有意义 |
| TROUT | 部分支持（渔业讨论略多），但活动记录过少，置信度低 | 对 6 人均可分析，但 trip 极少 |

### Q2：全量知识图谱下——委员会时间分配与整体是否偏袒？

**使用数据**：仅 `journalist.json`。

**时间分配的四个维度**：

| 维度 | 内容 | 可视化形式 |
|------|------|----------|
| 会议轴 | 16 次会议的议题产业结构 | 时间河流图（StreamGraph） |

| 实际出行 | trip 的时间 + 途经 place | 地图轨迹 + 时间轴 |

**Q2 图表**：
1. **议题时间河流图**：横轴=会议序号，纵轴=讨论量，颜色=产业分类（总览时间分配趋势）
2. **成员 × 产业 热力矩阵**：6 人对渔/游/混合议题的参与次数热力图（个人分配）
3. **地图 trip 落点**：按 zone 着色（工业/旅游/商业/政府），显示空间分配
4. **bias_index 仪表盘**：整体偏见指数 + 分成员偏见指数条形图

**必答结构**：
1. 时间分配事实（渔/游/混合各占比多少）
2. 是否偏袒（偏渔 / 偏游 / 均衡 / 转型期双重关注）
3. 区分「活动量偏见」与「情感偏见」（sentiment 均值可能与参与次数方向不同）

## 五、技术栈选型（与 BAIT 参考作品的差异化）

参考作品 BAIT 使用了 Vue 3 + FastAPI + **Neo4j 图数据库** + D3.js 的架构。我们的系统在以下方面做出差异化选择：

| 层次 | 我们的选择 | BAIT 的选择 | 差异化理由 |
|------|----------|-----------|----------|
| **数据处理** | Python + NetworkX + Pandas | Python + Neo4j | 避免依赖图数据库，离线预处理后输出 JSON，部署更简单 |
| **后端** | Python FastAPI（轻量，仅提供预处理数据） | FastAPI + Neo4j 实时查询 | 数据量有限，预处理后静态服务即可 |
| **前端框架** | Vue 3 + TypeScript + Vite | Vue 3 + TypeScript + Vite | 相同（通用框架，非原创） |
| **图表库** | **Apache ECharts** | D3.js | ECharts 声明式配置 + 丰富内置图表类型，开发效率更高 |
| **图谱可视化** | **AntV G6** | D3.js force-graph | G6 专为图谱设计，API 更高层，ego 网络交互更成熟 |
| **地图** | Leaflet.js + GeoJSON | D3.js + GeoJSON | Leaflet 的地图瓦片支持更直观的地理背景 |
| **状态管理** | Pinia | Pinia | 相同（Vue 3 标准选项） |
| **UI 组件** | **Ant Design Vue** | PrimeVue + Tailwind | 不同的组件库，风格更统一简洁 |
| **视觉隐喻** | **数据集对比雷达 + 偏见量表** | 倾斜天平（Scale） | 设计独立的视觉语言 |

### 5.1 数据预处理流程（Python 脚本）

